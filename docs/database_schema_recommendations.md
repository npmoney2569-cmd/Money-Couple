# Database Schema Recommendations & Future Improvements

เอกสารนี้รวบรวมข้อเสนอแนะและจุดที่สามารถปรับปรุงได้ในอนาคต สำหรับ Database Schema ของระบบบัญชีรายรับ-รายจ่าย (CMN) จากการวิเคราะห์โครงสร้างฐานข้อมูลปัจจุบัน

## 1. สถาปัตยกรรม Couple Mode และ Shared Accounts
ปัจจุบันระบบรองรับการแชร์ค่าใช้จ่าย (Splits & Settlements) คล้ายคลึงกับแอปพลิเคชัน Splitwise โดยที่ `accounts` และ `transactions` ยังคงผูกติดกับ `user_id` เพียงคนเดียว

**ข้อเสนอแนะในอนาคต:**
หากต้องการขยายระบบให้รองรับ **"บัญชีกองกลาง (Joint Bank Account)"** ที่ผู้ใช้หลายคนสามารถเป็นเจ้าของร่วมกันได้อย่างแท้จริง อาจจะต้องพิจารณาปรับปรุง Schema ดังนี้:
- เปลี่ยนจาก `user_id` ตรงๆ ไปเป็น `owner_id` (ซึ่งอาจจะเป็น `user_id` หรือ `couple_id` ก็ได้)
- หรือเพิ่ม `couple_id` เข้าไปในตาราง `accounts` และ `transactions` ควบคู่กัน เพื่อระบุความเป็นเจ้าของร่วม

## 2. การรองรับหลายสกุลเงิน (Multi-Currency)
แม้ว่าปัจจุบันตาราง `users` และ `accounts` จะมีการระบุ `currency` แล้ว แต่ยังขาดการจัดการสกุลเงินในระดับ Transaction

**ข้อเสนอแนะในอนาคต:**
หากผู้ใช้เดินทางไปต่างประเทศและมีการใช้จ่ายข้ามสกุลเงิน (เช่น บัญชีหลักเป็น THB แต่รูดบัตรเป็น USD) ระบบควรเก็บข้อมูลต้นทางไว้เพื่อความโปร่งใสในการตรวจสอบย้อนหลัง
- ควรเพิ่ม Field `original_amount`, `original_currency` (สกุลเงินที่ใช้จ่ายจริง) และ `exchange_rate` (อัตราแลกเปลี่ยน ณ วันที่ทำรายการ) ลงในตาราง `transactions`

## 3. ความสม่ำเสมอของการจัดการข้อมูล (Soft Deletes vs Active Status)
ตาราง `transactions` มีการออกแบบที่ดีมากโดยใช้ **Soft Delete** (`deleted_at`) ซึ่งช่วยให้สามารถทำฟีเจอร์ Undo และป้องกันปัญหาข้อมูลทางบัญชีเพี้ยนหรือสูญหายได้

**ข้อเสนอแนะในอนาคต:**
- ตารางอื่นๆ เช่น `accounts` และ `categories` ปัจจุบันใช้ Field `is_active` (Boolean) ในการซ่อน/ปิดการใช้งานข้อมูล
- หากผู้ใช้พยายามลบ Account ที่มี Transaction ผูกอยู่ ระบบจะติด Constraint Error (`RESTRICT` จาก Foreign Key)
- เพื่อความสม่ำเสมอ ควรพิจารณาเปลี่ยนไปใช้ **Soft Delete** (`deleted_at`) ในตาราง `accounts`, `categories` และ `tags` ด้วย เพื่อให้มีแนวทางการจัดการข้อมูลเหมือนกับ `transactions`

## 4. ประสิทธิภาพเมื่อข้อมูลมีขนาดใหญ่ (Performance & Archiving)
ตาราง `audit_logs` เป็นตารางที่ช่วยในการตรวจสอบย้อนหลังได้อย่างดีเยี่ยม แต่จะเป็นตารางที่มีจำนวน Row เพิ่มขึ้นอย่างรวดเร็วที่สุดในระบบ 

**ข้อเสนอแนะในอนาคต:**
- ควรวางแผนเรื่อง **Table Partitioning** (เช่น แบ่ง Partition ของ `audit_logs` ตามเดือนหรือปี)
- หรือทำ **Data Retention Policy (Cron Job)** เพื่อลบหรือ Backup ข้อมูล Audit Log ที่มีอายุเกินกำหนด (เช่น เก่ากว่า 1 ปี) ไปเก็บไว้ใน Cold Storage เพื่อรักษาประสิทธิภาพการอ่าน/เขียนของ Database ตัวหลัก
